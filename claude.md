# beauty-saas-crm 로컬 지침

## 프로젝트 범위

이 폴더는 카리스뷰티 beauty-saas-crm 프로젝트 전용이다.
다른 프로젝트(KSGI, 킹옥션, 블로그자동화, MG컴퍼니 등)의 규칙, 구조, 문구, 프롬프트를 섞지 않는다.

## 최우선 원칙

1. 전역지침보다 현재 폴더의 로컬 지침을 우선 적용한다.
2. 작업 전 반드시 현재 프로젝트 구조를 먼저 확인한다.
3. MCP 연결 정보가 있으면 먼저 확인하고 적용한다.
4. 구조 확인 없이 큰 수정부터 진행하지 않는다.
5. 필요한 파일만 최소 범위로 수정한다.
6. 기능 확장보다 안정화와 정합성을 우선한다.

## 현재 우선순위

현재 프로젝트는 기능 추가보다 기반 안정화가 우선이다.

우선순위:

1. .env 및 키 보안 점검
2. Customers.tsx 핵심 오류 복구
3. schema.ts 와 실제 사용 필드 정합성 맞추기
4. role / 권한 처리 문자열 일관성 유지
5. lint 최소 정리
6. 이후 로그인 편의성, 예약관리, 고객관리, 유료 SaaS 전환 기능 개선

## 현재 확인된 핵심 이슈

### 1. .env 보안

- .env가 git 추적 중인지 반드시 점검한다.
- 프론트엔드 코드에 SUPABASE_SERVICE_ROLE_KEY를 두지 않는다.
- 키 값 자체는 절대 출력하지 않는다.

### 2. Customers 화면 오류

- Customers.tsx의 상태값 정의 누락 여부를 먼저 확인한다.
- activeFilter, setActiveFilter, expandedId, setExpandedId 관련 오류를 최소 범위로 복구한다.

### 3. 타입 정합성

- usePermissions.ts에서 사용하는 필드(plan, ai_enabled, sms_count, sms_limit)와 schema.ts의 Shop 타입을 일치시킨다.
- SQL/migration 기준과 TS 타입 기준이 다르면 먼저 정합성을 맞춘다.

### 4. 권한 처리

- role 기준은 'admin' | 'staff'를 유지한다.
- 'ADMIN' 같은 대소문자 혼용은 허용하지 않는다.

## 기능 개선 우선 기준

안정화 이후 기능 개선 시 아래 순서를 우선 검토한다.

1. 로그인 / 자동로그인 / 저장 편의성
2. 예약 등록 흐름
3. 고객 검색 + 신규 등록 연결
4. 스케줄표 가독성
5. 대시보드 구조 정리
6. 유료 SaaS 전환 구조
7. 결제/요금제 관련 확장

## 작업 방식

항상 아래 순서로 진행한다.

1. 현재 작업 목표 확인
2. 관련 파일 확인
3. 로컬 지침 확인
4. MCP 연결 여부 확인
5. 영향 범위 확인
6. 최소 범위 수정
7. 수정 후 변경 이유와 영향 범위 요약

## 금지사항

1. 다른 프로젝트 설정을 현재 프로젝트에 섞지 않는다.
2. 근거 없이 구조를 크게 바꾸지 않는다.
3. UI를 먼저 예쁘게 바꾸는 작업을 우선하지 않는다.
4. 보안 이슈를 미룬 채 기능 추가부터 하지 않는다.
5. 타입 오류가 많은 상태에서 대규모 기능 확장을 하지 않는다.

## 보고 방식

1. 먼저 현재 상태를 짧게 진단한다.
2. 수정 전 목표와 범위를 분명히 한다.
3. 수정은 최소 범위로 진행한다.
4. 수정 후 바뀐 파일, 수정 이유, 영향 범위를 짧게 정리한다.
5. 키, 비밀값 등 민감한 값은 절대 출력하지 않는다.

## MCP 원칙

1. MCP 연결 정보가 있으면 일반 추측보다 MCP 기반 확인을 우선한다.
2. MCP 연결 정보가 없으면 현재 코드와 로컬 문서를 기준으로 판단한다.
3. MCP가 필요한 프로젝트라고 판단되면 먼저 이 로컬 지침에 연결 기준을 추가하는 방향을 검토한다.
